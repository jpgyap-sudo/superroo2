# Gap Audit - README, Monitoring, and Self-Healing Proof

Date: 2026-05-21  
Owner: SuperRoo product/agent workflow

## Executive Summary

The three user-reported gaps are real, but their shape has changed:

1. **README positioning was the biggest public-facing gap.** It still read like a generic/forked editor extension and hid the Central Brain, Telegram, cloud dashboard, learning layer, and self-healing story.
2. **Monitoring exists but proof surfaces are uneven.** There are monitoring APIs and dashboard views, but some endpoints still depended on JSON snapshots instead of the live orchestrator database.
3. **Self-healing has tests and primitives, but needs operational evidence.** Metrics exist in code, yet the public/operator story needs live success rates, repair execution counts, repeated-failure tracking, and escalation visibility.

## Gap 1 - README looked generic/fork-like

**Status after this pass:** Improved.

### What existed

- README emphasized generic editor-assistant behavior: code generation, modes, tutorials, extension setup.
- It contained release/community-handoff copy that made the repo look like a fork or continuation notice instead of a distinct platform.
- It did not lead with SuperRoo's differentiators: Central Brain, learning layer, Telegram, cloud dashboard, self-healing, unified deployment, and monitoring.

### Changes made

- Rewrote the top README positioning around SuperRoo as an autonomous AI engineering platform.
- Added tables/sections for Central Brain, learning layer, cloud dashboard, Telegram control plane, self-healing, and unified deployment.
- Kept local setup, disclaimer, contributing, and license sections intact.

### Remaining gaps

- Add screenshots/GIFs of the cloud dashboard and Telegram operator flow.
- Add a short architecture diagram near the top.
- Link to a stable hosted demo or recorded walkthrough when available.

## Gap 2 - Monitoring still incomplete

**Status after this pass:** Partially improved; still needs alerting proof.

### What exists

- `cloud/api/routes/monitoring.js` exposes log, stats, health timeline, service, and aggregated-log endpoints.
- `cloud/dashboard/src/components/views/monitoring.tsx` provides a monitoring dashboard.
- `cloud/api/routes/healing-metrics.js` exposes healing metrics, incidents, and escalated incident endpoints.

### Gaps found

- Healing proof endpoints were biased toward JSON fallback files, which can diverge from live orchestrator state.
- Monitoring is mostly pull-based dashboard/API visibility; there is no clear alert-rule layer for incident-rate spikes or repeated healing failures.
- Data-source transparency was weak, making it hard to know whether an operator was looking at live DB data or fallback snapshots.

### Changes made

- Updated `cloud/api/routes/healing-metrics.js` to prefer live data in this order:
    1. `global.__orchestrator.healingBus` when available,
    2. SQLite at `ORCHESTRATOR_DB_PATH`,
    3. SQLite at `cloud/orchestrator/data/orchestrator.db`,
    4. JSON fallback files.
- Added `dataSource` to responses so dashboards can display whether metrics are live or fallback.
- Added schema detection via `PRAGMA table_info` so both known `healing_incidents` schemas can be read defensively.

### Remaining gaps

- Add alert thresholds and Telegram notifications for:
    - high recent error rate,
    - repeated healing failures,
    - escalating open incidents,
    - missing or stale metrics source.
- Consider Server-Sent Events or WebSocket streaming for healing/monitoring updates.
- Add tests that exercise both SQLite healing schemas and JSON fallback behavior.

## Gap 3 - Self-healing needs real-world proof

**Status after this pass:** Proof metrics are better exposed; real-world case studies still needed.

### What exists

- `src/super-roo/healing/HealingMetrics.ts` tracks success/failure rates, rolling trends, and classification metrics.
- `src/super-roo/healing/SelfHealingLoop.ts` tracks repair attempts, per-category escalation, notification routes, and circuit breaker signals.
- Healing bus tests cover repair tracking, success rates, escalation, and repeated failures.

### Gaps found

- Operational dashboards needed to expose repair execution/action counts and repeated-failure signals from live DB rows.
- Escalated incidents needed a consistent endpoint surface across old/new schemas.
- Real-world repair examples and production success-rate snapshots are not yet documented.

### Changes made

- Healing metrics responses now include:
    - `repairExecutions`,
    - `repairActions`,
    - `escalationCount`,
    - `repeatedFailures`,
    - `recentTrend`,
    - `dataSource`.
- Incidents and escalated endpoints now prefer live source data and normalize known schema variants.

### Remaining gaps

- Record and publish anonymized case studies for successful repairs.
- Add a repair audit timeline endpoint for each incident.
- Add dashboard badges for stale metrics/fallback mode.
- Track success rate by repair plan type and category from production runs over time.

## Recommended Next Actions

1. Add tests for `cloud/api/routes/healing-metrics.js` covering orchestrator, SQLite, and JSON fallback modes.
2. Add Telegram alert routing for repeated-failure and escalation thresholds.
3. Add a dashboard banner when `dataSource !== "orchestrator" && dataSource !== "sqlite"`.
4. Create a self-healing case-study doc once real production repair samples are available.
